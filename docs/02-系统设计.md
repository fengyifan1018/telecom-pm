### 架构说明

采用**单体优先**策略（Modular Monolith）：
- 初期：单体应用，模块化内部结构，降低部署运维复杂度
- 演进：当某模块（如通知服务）压力大时，可独立拆出为微服务
- 理由：50-200人规模，微服务的运维成本远大于收益

## 2. 技术栈推荐

| 层次 | 推荐方案 | 理由 |
|------|----------|------|
| 后端 | Python + FastAPI | 开发效率高，async支持好，团队易招人 |
| 前端 | Vue 3 + Element Plus | 国内生态好，组件丰富，适合管理后台 |
| 移动端 | uni-app 或 企微H5 | 一套代码多端，降低维护成本 |
| 数据库 | PostgreSQL | 支持JSON字段（存灵活配置），性能可靠 |
| 缓存 | Redis | 会话管理、任务状态缓存、消息队列 |
| 文件存储 | MinIO（自建）或 OSS（云） | 施工照片、文档、验收报告 |
| 消息推送 | 企业微信 API / WebSocket | 任务通知实时到达 |

备选方案：如果团队 Java 能力更强，可选 Spring Boot + MyBatis-Plus 替代后端。

### 3.2 核心表结构

#### project（项目）
```sql
CREATE TABLE project (
    id              BIGSERIAL PRIMARY KEY,
    project_no      VARCHAR(32) UNIQUE NOT NULL,   -- 项目编号（如 PRJ-2026-0001）
    name            VARCHAR(200) NOT NULL,
    customer_id     BIGINT NOT NULL,
    product_type    VARCHAR(20) NOT NULL,           -- transmission/dark_fiber/dia/sdwan
    status          VARCHAR(20) NOT NULL DEFAULT 'draft',
    priority        SMALLINT DEFAULT 3,            -- 1-5
    sales_id        BIGINT NOT NULL,               -- 销售
    pm_id           BIGINT,                        -- 项目经理
    contract_amount DECIMAL(12,2),
    planned_start   DATE,
    planned_end     DATE,
    actual_start    DATE,
    actual_end      DATE,
    description     TEXT,
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);
```

#### task（任务）
```sql
CREATE TABLE task (
    id              BIGSERIAL PRIMARY KEY,
    project_id      BIGINT NOT NULL REFERENCES project(id),
    parent_task_id  BIGINT REFERENCES task(id),    -- 支持子任务
    task_no         VARCHAR(32) UNIQUE NOT NULL,
    title           VARCHAR(200) NOT NULL,
    description     TEXT,
    phase           VARCHAR(30) NOT NULL,           -- 阶段标识
    status          VARCHAR(20) NOT NULL DEFAULT 'pending',
    assignee_id     BIGINT,                        -- 负责人
    reviewer_id     BIGINT,                        -- 审核人
    priority        SMALLINT DEFAULT 3,
    planned_start   DATE,
    planned_end     DATE,
    actual_start    DATE,
    actual_end      DATE,
    created_at      TIMESTAMP DEFAULT NOW(),
    updated_at      TIMESTAMP DEFAULT NOW()
);
```

#### task_dependency（任务依赖）
```sql
CREATE TABLE task_dependency (
    id              BIGSERIAL PRIMARY KEY,
    task_id         BIGINT NOT NULL REFERENCES task(id),
    depends_on_id   BIGINT NOT NULL REFERENCES task(id),
    dependency_type VARCHAR(10) DEFAULT 'finish_to_start' -- FS/SS/FF/SF
);
```

#### task_transition（任务流转记录）
```sql
CREATE TABLE task_transition (
    id              BIGSERIAL PRIMARY KEY,
    task_id         BIGINT NOT NULL REFERENCES task(id),
```

### 3.3 任务状态机

```
                    ┌─── 退回 ───┐
                    ▼             │
┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
│ pending │───→│ active │───→│ review │───→│  done  │
│  待分配  │    │ 进行中  │    │ 待审核  │    │ 已完成  │
└────────┘    └────────┘    └────────┘    └────────┘
     │                                         │
     └───────────── 取消 ──────────────────────→ closed
```

状态说明：
- **pending**：任务已创建，等待分配/接受
- **active**：负责人已接受，正在执行
- **review**：执行完毕，等待审核确认
- **done**：审核通过，任务完成
- **closed**：任务取消或作废

## 4. 任务流转引擎设计

### 4.1 流转规则

```python
# 伪代码：任务完成后的自动流转逻辑
def on_task_completed(task):
    # 1. 检查同阶段是否还有未完成的必需任务
    phase_tasks = get_required_tasks(task.project_id, task.phase)
    if any(t.status != 'done' for t in phase_tasks if t.id != task.id):
        return  # 本阶段还有任务未完成

    # 2. 查找依赖当前阶段的下一批任务
    next_tasks = get_tasks_depending_on(task.phase)
    
    for next_task in next_tasks:
        # 3. 检查该任务的所有前置依赖是否都已满足
        if all_dependencies_met(next_task):
            # 4. 激活任务并通知负责人
            activate_task(next_task)
            notify_assignee(next_task)
```

### 4.2 通知策略

| 事件 | 通知对象 | 通知方式 |
|------|----------|----------|
| 任务分配给我 | 负责人 | 企微消息 + 站内通知 |
| 任务被退回 | 原负责人 | 企微消息 + 站内通知 |
| 任务超期 | 负责人 + 项目经理 | 企微消息 |
| 即将超期（提前1天） | 负责人 | 站内通知 |
| 阶段完成 | 项目经理 | 站内通知 |
| 项目交付完成 | 销售 + 项目经理 | 企微消息 |

### 4.3 退回机制

退回时必须填写原因，系统记录完整流转历史。退回规则：
- 只能退回到上一个相关阶段（不能跳阶段退回）
- 退回后任务状态重置为 active，原负责人收到通知
- 退回次数纳入统计，用于识别流程瓶颈

# 通信项目任务管理系统 - 异常场景设计

## 概述

正常流转路径只覆盖约60-70%的实际情况。本文档定义系统如何处理偏离正常路径的场景。

---

## 1. 项目暂停与恢复

### 1.1 触发条件

| 场景 | 触发人 | 典型原因 |
|------|--------|----------|
| 客户要求暂停 | 销售/项目经理 | 客户预算冻结、内部审批未通过、选址变更 |
| 资源不可用 | 运营 | 关键资源被占、上游运营商无法提供 |
| 外部阻塞 | 项目经理 | 施工许可未批、物业不配合、不可抗力 |
| 商务问题 | 销售 | 合同条款争议、付款问题 |

### 1.2 暂停处理逻辑

```
项目暂停时系统行为:
┌─────────────────────────────────────────────┐
│ 1. 项目状态 → suspended                     │
│ 2. 所有进行中任务 → paused (冻结计时)        │
│ 3. 所有待分配任务 → 暂不激活                  │
│ 4. 通知所有相关人员                           │
│ 5. 记录暂停原因、预计恢复时间                  │
│ 6. 释放已锁定但未使用的资源(可选)              │
└─────────────────────────────────────────────┘
```

### 1.3 数据结构

```sql
CREATE TABLE project_suspension (
    id              BIGSERIAL PRIMARY KEY,
    project_id      BIGINT NOT NULL REFERENCES project(id),
    suspended_by    BIGINT NOT NULL,
    suspend_reason  VARCHAR(20) NOT NULL,    -- customer_request/resource_unavailable/external_block/commercial
    description     TEXT,
    expected_resume DATE,
    suspended_at    TIMESTAMP DEFAULT NOW(),
    resumed_at      TIMESTAMP,
    resumed_by      BIGINT,
    resume_note     TEXT
);
```

### 1.4 恢复流程

```
恢复时系统行为:
┌─────────────────────────────────────────────┐
│ 1. 项目状态 → active                        │
│ 2. paused 任务 → 恢复为原状态                │
│ 3. 自动重算截止时间(+暂停天数)                │
│ 4. 通知所有相关人重新确认当前状态              │
│ 5. 项目经理确认: 是否需要重新做资源锁定        │
└─────────────────────────────────────────────┘
```

**截止时间自动顺延规则：**
- 暂停期间的天数不计入工期
- 恢复后所有未完成任务的 planned_end 自动顺延
- 项目整体交付时间同步更新
- 顺延后若超出客户承诺时间，自动标记预警

## 2. 需求变更

### 2.1 变更分级

| 级别 | 定义 | 示例 | 审批流程 |
|------|------|------|----------|
| 轻微 | 不影响方案和工期 | 联系人变更、备注修改 | 项目经理直接处理 |
| 一般 | 影响部分任务但不影响整体方案 | 带宽升级(10G→20G)、IP数量调整 | 项目经理审批 |
| 重大 | 需要重做方案或影响工期 | 路由变更、新增站点、产品类型变更 | 项目经理 + 运营审批 |

### 2.2 变更流程

```
         ┌──────┐
         │ 发起  │  销售/项目经理 填写变更单
         └──┬───┘
            ▼
     ┌──────────────┐
     │ 影响评估      │  项目经理评估对进度/成本/资源的影响
     │ (1个工作日内) │
     └──────┬───────┘
            ▼
    ┌───────────────┐
    │ 影响大？       │
    └───┬───────┬───┘
     是 │       │ 否
        ▼       ▼
  ┌──────────┐  ┌──────────┐
  │ 上级审批  │  │ PM直接批准│
  │ (运营总监)│  └─────┬────┘
  └─────┬────┘        │
        │ 批准         │
        ▼              ▼
  ┌──────────────────────┐
  │ 执行变更              │
  │ - 调整任务/新增任务    │
  │ - 更新排期            │
  │ - 通知相关人          │
  └──────────────────────┘
```

### 2.3 变更单数据结构

```sql

    action          VARCHAR(10) NOT NULL,         -- add/modify/remove/redo
```

### 2.4 变更对进行中任务的影响

**已完成的任务需要重做：**
```
方案设计(已完成) → 需求变更(路由变更) → 创建新任务"方案修订"
                                        ├─ 关联原任务
                                        ├─ 标记变更单号
                                        └─ 后续依赖任务自动重置为 pending
```

**进行中的任务需要调整：**
- 保留任务但更新描述/要求
- 在任务评论中自动插入变更说明
- 通知负责人查看新要求

## 3. 资源冲突

### 3.1 冲突类型

| 类型 | 说明 | 发生阶段 |
|------|------|----------|
| 端口占用 | 规划的端口被其他项目抢先使用 | 资源确认/施工 |
| 光纤资源不足 | 计划路由的管道/光缆容量满 | 资源确认 |
| 人员冲突 | 同一工程师同时被分配多个项目 | 任务分派 |
| 设备缺货 | 采购的设备供应商无货 | 采购 |
| 施工窗口冲突 | 客户机房维护窗口被占 | 施工实施 |

### 3.2 人员冲突处理

**检测机制：**
```python
def check_assignee_conflict(user_id, planned_start, planned_end):
    """分配任务前检查人员时间冲突"""
    overlapping = Task.query.filter(
        assignee_id=user_id,
        status__in=['active', 'pending'],
        planned_start <= planned_end,
        planned_end >= planned_start
    ).all()
    
    if len(overlapping) >= CONCURRENT_TASK_LIMIT:  # 默认3
        return ConflictWarning(
            user=user_id,
            existing_tasks=overlapping,
            message=f"该人员同期已有{len(overlapping)}个任务"
        )
    return None
```

**处理方式：**
- 项目经理收到冲突预警，非阻塞（可强制分配）
- 仪表盘展示人员负荷热力图
- 建议替代人选（同角色、同期空闲）

### 3.3 资源抢占处理

场景：项目A和项目B同时需要同一个端口资源

```
┌─────────────────────────────────────────────────────┐
│ 资源冲突处理流程                                      │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 资源确认阶段发现冲突 → 系统标记为"资源冲突"        │
│                                                     │
│  2. 自动通知:                                        │
│     - 两个项目的项目经理                              │
│     - 运营(资源协调人)                               │
│                                                     │
│  3. 运营介入协调:                                    │
│     a) 优先级高的项目优先获得资源                      │
│     b) 寻找替代资源给另一个项目                       │
│     c) 若无替代 → 低优先级项目进入等待                 │
│                                                     │
│  4. 决策记录:                                        │
│     - 记录协调结果和原因                              │
│     - 受影响项目自动调整排期                          │
│     - 通知受影响项目的销售(可能影响客户承诺)            │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 4. 任务退回与返工

### 4.1 退回规则

```
谁可以退回:
- 审核人 → 退回给执行人 (质量不合格)
- 后续阶段 → 退回给前序阶段 (前序产出有问题)
- 项目经理 → 退回任何阶段 (协调需要)

退回限制:
- 必须填写退回原因(最少20字)
- 单任务最多退回3次，超过需项目经理介入
- 退回不能跨两个以上阶段(A→C 不允许直接退回到A，需逐级)
```

### 4.2 退回对后续任务的影响

```
场景: "方案设计"已完成 → "采购"已开始 → 发现方案有误需退回

系统行为:
┌─────────────────────────────────────────────────────┐
│ 1. "方案设计"状态 → rework                           │
│ 2. 检查依赖"方案设计"且已启动的任务                     │
│    → "采购"状态 → blocked (阻塞，等待方案修订)         │
│ 3. 通知采购人员: 方案修订中，暂停采购动作               │
│ 4. 方案修订完成后:                                    │
│    → 系统检查修订内容是否影响采购                      │
│    → 若影响: 采购任务需更新需求后恢复                  │
│    → 若不影响: 采购任务自动恢复为 active               │
└─────────────────────────────────────────────────────┘
```

### 4.3 返工统计

系统自动统计用于质量改进：
- 各阶段退回率
- 退回原因分类（设计不完整、信息缺失、质量不达标、需求理解偏差）
- 退回导致的工期延误累计
- 高退回率人员/环节识别

## 5. 项目取消

### 5.1 取消流程

```
发起取消(销售/PM) → 影响评估 → 审批 → 执行取消
                      │
                      ▼
              ┌─────────────────┐
              │ 评估内容:        │
              │ - 已产生的成本    │
              │ - 已采购的设备    │
              │ - 已占用的资源    │
              │ - 在途采购单     │
              └─────────────────┘
```

### 5.2 取消时的资源回收

```python
def cancel_project(project_id, reason, operator_id):
    project = get_project(project_id)
    
    # 1. 终止所有进行中的任务
    active_tasks = get_active_tasks(project_id)
    for task in active_tasks:
        task.status = 'closed'
        task.close_reason = 'project_cancelled'
        notify(task.assignee_id, 'task_cancelled', task)
    
    # 2. 取消待处理的采购单
    pending_orders = get_pending_purchase_orders(project_id)
    for order in pending_orders:
        if order.status == 'ordered':
            flag_for_manual_handling(order)  # 已下单的需人工处理
        else:
            order.status = 'cancelled'
    
    # 3. 释放锁定的资源
    locked_resources = get_locked_resources(project_id)
    for resource in locked_resources:
        release_resource(resource)
    
    # 4. 更新项目状态
    project.status = 'cancelled'
    project.cancel_reason = reason
    
    # 5. 通知所有参与人
    notify_all_participants(project_id, 'project_cancelled')
```

## 6. 超时升级机制

### 6.1 升级链路

```
任务超期 → Level 1: 通知负责人 (超期当天)
        → Level 2: 通知负责人 + 项目经理 (超期2天)
        → Level 3: 通知运营总监 (超期5天)
        → Level 4: 自动暂停项目，要求重新排期 (超期10天)
```

### 6.2 配置化升级规则

```sql
CREATE TABLE escalation_rule (

    trigger_type    VARCHAR(20) NOT NULL,    -- overdue/no_response/blocked

    action          VARCHAR(20) NOT NULL,    -- notify/escalate/suspend
    notify_roles    VARCHAR[] NOT NULL,      -- 通知角色列表
    is_active       BOOLEAN DEFAULT TRUE
);
```

默认规则：
```json
[
  {"trigger": "overdue", "days": 0, "action": "notify", "roles": ["assignee"]},
  {"trigger": "overdue", "days": 2, "action": "notify", "roles": ["assignee", "pm"]},
  {"trigger": "overdue", "days": 5, "action": "escalate", "roles": ["pm", "ops_director"]},
  {"trigger": "no_response", "days": 1, "action": "notify", "roles": ["assignee", "pm"]},
```

## 7. 并发操作冲突

### 7.1 场景

- 两人同时编辑同一任务
- 项目经理分配任务的同时，负责人在退回
- 批量操作与单条操作冲突

### 7.2 解决策略

采用**乐观锁 + 冲突提示**：

```sql
-- 每个实体增加版本号
ALTER TABLE task ADD COLUMN version INT DEFAULT 1;
ALTER TABLE project ADD COLUMN version INT DEFAULT 1;
```

```python
def update_task(task_id, updates, expected_version):
    result = db.execute("""
        UPDATE task 
        SET ..., version = version + 1, updated_at = NOW()
        WHERE id = :task_id AND version = :expected_version
    """)
    
    if result.rowcount == 0:
        # 版本冲突，说明被其他人修改了
        current = get_task(task_id)
        raise ConflictError(
            message="任务已被其他人修改，请刷新后重试",
            current_state=current,
            your_changes=updates
        )
```

前端处理：弹窗提示冲突，展示当前最新状态，用户决定是否覆盖。


```
- timeout 后自动升级通知项目经理
- 项目经理决策: 继续等待 / 启动备选方案 / 升级协调
```

## 9. 异常场景与正常流程的整合

### 状态机扩展

在原有状态基础上增加异常状态：

```
正常:    pending → active → review → done → closed
                     │         │
异常:                ▼         ▼
              ┌─ paused (项目暂停导致)
              ├─ blocked (资源冲突/外部依赖)
              ├─ rework (被退回返工)
              └─ cancelled (项目取消)

恢复路径:
  paused    → active (项目恢复)
  blocked   → active (阻塞解除)
  rework    → active (修改后重新提交)
  cancelled → [终态]
```

### 任务表补充字段

```sql
ALTER TABLE task ADD COLUMN block_reason TEXT;
ALTER TABLE task ADD COLUMN block_type VARCHAR(20);       -- resource/dependency/rework
ALTER TABLE task ADD COLUMN rework_count INT DEFAULT 0;
ALTER TABLE task ADD COLUMN original_planned_end DATE;    -- 记录初始截止(用于统计延误)
```

# 通信项目任务管理系统 - 协作流转详细设计

## 1. 跨角色协作核心场景

### 场景一：销售 → 项目经理（项目启动）

**触发条件：** 销售提交项目创建表单

**信息传递：**
- 客户名称、联系方式
- 产品类型、技术规格需求
- 合同金额、承诺交付时间
- 客户侧对接人

**系统行为：**
1. 自动创建项目，状态=draft
2. 通知指定项目经理（或由运营分配）
3. 项目经理确认接手 → 状态变为 active
4. 系统根据产品类型加载流程模板，生成任务树

**关键设计：** 销售录入时引导必填字段，避免信息不全导致后续反复沟通。

---

### 场景二：项目经理 → 各角色（任务分派）

**操作：** 项目经理在任务看板中指定负责人

**系统行为：**
1. 任务状态 pending → active
2. 企微推送通知负责人，含：任务标题、所属项目、截止时间
3. 负责人点击通知可直接跳转任务详情页
4. 若负责人24h未查看，升级通知项目经理

**退回场景：** 负责人可拒绝接受（说明原因），项目经理重新分配。

---

### 场景三：网络工程师 → 采购（采购需求）

**触发条件：** 方案设计阶段完成，输出设备清单

**信息传递：**
- 设备型号、数量
- 配置要求
- 期望到货时间
- 到货地址

**系统行为：**
1. 方案设计任务完成时，关联的采购任务自动激活
2. 采购任务中自动附带方案设计产出的设备清单
3. 采购人员收到通知，进入任务查看需求
4. 采购更新预计到货时间 → 系统自动调整后续任务排期

---

### 场景四：现场实施 → 网络工程师（问题升级）

**触发条件：** 现场实施遇到方案与实际不符

**操作流程：**
1. 现场人员在任务中发起"问题升级"
2. 填写问题描述、现场照片
3. 系统通知网络工程师
4. 网络工程师回复解决方案或决定退回重新设计
5. 问题解决后恢复任务执行

**数据结构：**
```sql
CREATE TABLE task_escalation (
    id              BIGSERIAL PRIMARY KEY,
    task_id         BIGINT NOT NULL REFERENCES task(id),
    raised_by       BIGINT NOT NULL,
    assigned_to     BIGINT NOT NULL,
    severity        VARCHAR(10) NOT NULL,   -- low/medium/high/critical
    title           VARCHAR(200) NOT NULL,
    description     TEXT,
    status          VARCHAR(20) DEFAULT 'open',  -- open/in_progress/resolved
    resolution      TEXT,
    created_at      TIMESTAMP DEFAULT NOW(),
    resolved_at     TIMESTAMP
);
```

---

### 场景五：验收流程（多方确认）

**参与方：** 网络工程师（测试）+ 现场实施（陪同）+ 客户（签字）+ 项目经理（确认）

**系统行为：**
1. 验收任务到达 → 通知网络工程师安排验收
2. 网络工程师上传测试报告
3. 现场人员上传客户签字照片/电子签
4. 项目经理最终审核确认
5. 所有确认完成 → 项目进入结项阶段

---

## 2. 流转引擎详细设计

### 2.1 引擎核心概念

```
流程模板 (WorkflowTemplate)
    └─ 阶段 (Phase) [有序列表]
         ├─ 依赖关系 (depends_on)
         ├─ 默认角色 (role)
         └─ 任务模板 (TaskTemplate) [列表]
              ├─ 是否必需 (required)
              ├─ 默认负责角色 (role)
              └─ 预估工期 (estimated_days)
```

### 2.2 自动流转规则

```python
class TransitionEngine:
    """任务流转引擎"""
    
    def complete_task(self, task_id, operator_id, result_data=None):
        task = self.get_task(task_id)
        
        # 1. 更新任务状态
        task.status = 'done'
        task.actual_end = now()
        self.log_transition(task, 'review', 'done', operator_id)
        
        # 2. 检查阶段完成情况
        phase_complete = self.check_phase_completion(task.project_id, task.phase)
        
        if phase_complete:
            # 3. 触发后续阶段
            self.activate_next_phases(task.project_id, task.phase)
            
            # 4. 更新项目整体进度
            self.update_project_progress(task.project_id)
    
    def check_phase_completion(self, project_id, phase):
        """检查某阶段是否所有必需任务已完成"""
        required_tasks = Task.query.filter(
            project_id=project_id, 
            phase=phase, 
            is_required=True
        ).all()
        return all(t.status == 'done' for t in required_tasks)
    
    def activate_next_phases(self, project_id, completed_phase):
        """激活依赖当前阶段的后续阶段"""
        project = self.get_project(project_id)
        template = self.get_template(project.product_type)
        
        for phase_def in template.phases:
            if completed_phase in phase_def.get('depends_on', []):
                # 检查该阶段的所有依赖是否都满足
                all_deps_met = all(
                    self.check_phase_completion(project_id, dep)
                    for dep in phase_def['depends_on']
                )
                if all_deps_met:
                    self.activate_phase_tasks(project_id, phase_def)
    
    def activate_phase_tasks(self, project_id, phase_def):
        """激活某阶段的所有任务"""
        tasks = Task.query.filter(
            project_id=project_id, 
            phase=phase_def['phase'],
            status='pending'
        ).all()
        
        for task in tasks:
            task.status = 'active'
            if task.assignee_id:
                self.notify(task.assignee_id, 'task_activated', task)
```

### 2.3 超时处理

```python
# 定时任务（每小时执行）
def check_overdue_tasks():
    overdue = Task.query.filter(
        status='active',
        planned_end < today(),
        overdue_notified=False
    ).all()
    
    for task in overdue:
        # 通知负责人
        notify(task.assignee_id, 'task_overdue', task)
        # 通知项目经理
        project = get_project(task.project_id)
        notify(project.pm_id, 'task_overdue_escalation', task)
        # 标记已通知
        task.overdue_notified = True

# 提前预警（每天早上9点）
def check_upcoming_due():
    upcoming = Task.query.filter(
        status='active',
        planned_end == today() + 1day
    ).all()
    
    for task in upcoming:
        notify(task.assignee_id, 'task_due_tomorrow', task)
```

## 3. 信息传递设计

### 3.1 上下文接力

核心问题：后一阶段的人如何获取前一阶段的产出？

**解决方案：阶段交付物（Deliverable）**

```sql
CREATE TABLE phase_deliverable (
    id              BIGSERIAL PRIMARY KEY,
    project_id      BIGINT NOT NULL REFERENCES project(id),
    phase           VARCHAR(30) NOT NULL,
    title           VARCHAR(200) NOT NULL,
    content_type    VARCHAR(20),         -- document/checklist/form
    content         JSONB,               -- 结构化数据
    attachment_ids  BIGINT[],            -- 关联附件
    submitted_by    BIGINT NOT NULL,
    submitted_at    TIMESTAMP DEFAULT NOW()
);
```

**典型交付物：**
- 勘察阶段 → 勘察报告（照片、环境信息、路由方案）
- 设计阶段 → 技术方案（设备清单、配置方案、拓扑图）
- 采购阶段 → 采购结果（到货时间、发货信息）
- 实施阶段 → 施工记录（安装照片、测试数据）
- 验收阶段 → 验收报告（测试结果、客户签字）

后续阶段的任务详情页自动展示前序阶段的交付物摘要。

## 4. 通知系统设计

### 4.1 通知渠道优先级

```
紧急（超时/Critical升级）  → 企微消息 + 电话提醒
重要（任务分配/退回）       → 企微消息 + 站内通知
一般（评论/进度更新）       → 站内通知
低优先（统计报表）          → 日报/周报邮件
```

### 4.2 通知聚合

避免消息轰炸：
- 同一项目的多条通知，5分钟内聚合为一条
- 用户可设置"免打扰"时段
- 非紧急通知按批次推送（每30分钟）

### 4.3 消息模板

```
【任务分配】
项目：{project_name}
任务：{task_title}
截止：{due_date}
分配人：{assigner_name}
点击查看详情 →

---

【任务超期预警】
⚠️ 以下任务已超期：
项目：{project_name}
任务：{task_title}
负责人：{assignee_name}
计划完成：{planned_end}
已超期：{overdue_days}天
```

## 5. 移动端适配（现场人员重点）

### 5.1 现场人员核心操作

| 操作 | 说明 | 交互方式 |
|------|------|----------|
| 查看我的任务 | 待办任务列表 | 列表 + 筛选 |
| 更新任务状态 | 开始/完成 | 一键操作 |
| 拍照上传 | 施工记录、环境照片 | 调用相机 |
| 填写表单 | 勘察报告、施工记录 | 结构化表单 |
| 问题上报 | 现场异常升级 | 表单 + 拍照 |
| 获取施工规范 | 查看操作指引 | 文档查看 |
| 客户签字 | 电子签名 | 手写签名板 |

### 5.2 离线支持

现场可能网络不佳：
- 任务列表本地缓存
- 照片先存本地，恢复网络后自动上传
- 表单数据暂存草稿
